内存池结构则如下所示，值得关注的有如下几点：

  1. ● 衍生 LRU 算法：midpoint 技术

MySQL 通过 information_scheme 和 show variables like 等语句来展示缓冲状态等 及相关统计信息，以供监视、调试、调优使用。详见书本内容。
fig: InnoDB 内存数据对象

为优化性能，缓存挡住了绝大部分流量，写先到缓存再到磁盘，读直接到缓存。为达到原子 性持久性：必须持久化和使用 WAL. 同理为优化性能，WAL 也并非实时更新到硬盘，也使用 缓冲策略（缓存缓冲本质都是先让内存挡一下的 memory hierachy）。那么很自然的问题就 是面对数据的增长，WAL 多久做一次 checkpoint, 做 checkpoint 时丢弃 WAL log 时与 checkpint 的数据保持一致性。

1.2.2. Checkpoint

Checkpoint 所做的事情无外乎是将缓冲池中的脏页刷回到磁盘，有如下要点需要考虑：(a) 每次刷新多少页到磁盘，(b) 每次从哪里取脏页，(c) 以及什么时间触发 Checkpoint

InnoDB CheckPoint 的两种方式

  • SharpCheckpoint 发生在数据库关闭时将所有的脏页都刷新回磁盘，但是对性能有影响，会 STW 吗？
  • FuzzyCheckpoint 页的刷新，即只刷新一部分脏页，而不是刷新所有的脏页回磁盘。

InnoDB 存储引擎中可能发生如下几种情况的 Fuzzy Checkpoint：

  • Master Thread Checkpoint
  • FLUSH_LRU_LIST Checkpoint
  • Async/Sync Flush Checkpoint
  • Dirty Page too much Checkpoint
fig: InnoDB Architecture

在 InnoDB 结构中，上述的 checkpoint 是对于缓冲区的持久化阐述。属于内存部分。而结 构中另一部分 Master Thread 工作方式具体是怎样的将在下面得到阐述。

1.2.4. InnoDB 关键特性

(1) 插入缓冲（提高性能）

假如一个表存在 random-distributed secondary index 的话呢，理论上写到磁盘的 random access 就会引起严重的 IO 性能问题。虽然实际情况可能会有的相对没那么严重， 例如用户购买时间是一个 secondary index, 时间总是增长的，该索引插入时则为 sequential access. MySQL 解决 random-distributed secondary index 该问题使用到一 个通用的技术：buffer + batch process, 具象到存储访问，即为 transform random access in memory to sequential access in disk, 该技术在实现上采用仍然 B+树 作为 insert buffer 的数据结构。

该 Insert Buffer 的方法实现细节要点有：(a) 数据结构的管理字段设计；(b) 没有缓冲 索引页怎么办，何时 merge 到 disk 的索引页，即包含 merge mutliple random-accessed page and write to disk.

InnoDB’s LRU 算法中的 midpoint 技术。当新插入数据，查到 mipoint 处，防止"扫描 一类"的任务污染缓存，导致过多热点数据失效。其中变量 innodb_old_blocks_time 规定新数据假如到 midpoint 后何时移到 热点数据部分。scan 一类任务的插入后经 过一段时间后再移到新数据部分不照样污染缓存吗, 此外通过观察 SHOW ENGINE INNODB STATUS 的结果得知因为 没有改变 innodb_old_blocks_time 的值，所以 page not made young 的值为 0. 所有的数据首先插到 midpoint 位置，此时 new 部分和 midpoint 这里的准热点数据都存于内存中，而且并没有因为 scan 任务的频繁加载表导致 真热点数据被淘汰，一段时间过去后， 有的准热点数据成功熬过了 innodb_old_blocks_time 转正，发生了page made young，有的准热点数据被新来的数据踢走，即 page not made young.
